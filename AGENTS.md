# SpectraDB Agent 工作说明 最近修改：21:55 7/2/2026

## 1. 项目定位

SpectraDB 是一个面向有机小分子多模态光谱分析的数据库建设项目。

项目目标是构建一个可靠、可扩展、可复现的有机小分子光谱数据库，为后续多模态光谱融合、未知样品识别、开放集识别和模型可解释性分析提供数据基础。

核心关联标识包括：

- CID
- SMILES
- InChIKey
- CAS
- 分子式
- 数据源内部 ID

当前重点光谱模态包括：

- IR
- Raman
- UV-Vis

后续可扩展模态包括：

- NMR
- MS

当前重点数据源包括：

- PubChem
- NIST WebBook
- SDBS

后续可扩展数据源包括：

- HMDB
- MassBank
- 其他公开机器可读光谱数据库

---

## 2. 当前项目阶段

当前处于：

**Phase 1：数据获取基础设施建设阶段。**

当前阶段的首要目标不是追求最大数据量，也不是追求严格随机抽样，而是建立稳定、可靠、可重复的数据获取流程。

当前阶段的完成标准：

1. NIST 能够稳定下载真实 JCAMP-DX 文件。
2. SDBS 能够稳定识别命中页面、详情页和真实光谱入口。
3. CAS 缓存能够稳定工作。
4. 所有采集脚本支持断点续跑。
5. 所有采集结果经过真实性校验。
6. 不再把 HTML、空页面、错误页或占位文件误判为光谱数据。
7. 小样本测试通过后，才能逐步扩大样本规模。

在 Phase 1 完成前，禁止盲目扩大采集规模。

---

## 3. 当前数据策略

当前默认采用：

**顺序采样 in-order sampling。**

当前默认输入优先级：

1. `master/compound_master_100000_in_order_clean.csv`
2. 其他明确指定的顺序采样 master

暂不优先使用：

- `compound_master_100000_random_clean.csv`
- 大规模随机采样数据
- 分层随机采样数据

原因：

当前阶段优先提高数据源覆盖率和采集成功率，而不是统计随机性。

随机采样将在 Phase 2 重新启用，用于评估分布偏差、泛化能力和开放集性能。

---

## 4. 数据真实性原则

数据真实性优先级高于数据规模。

必须遵守：

1. 不允许把 HTML 页面、错误页、免责声明页面、空页面保存为 `.jdx` 光谱文件。
2. 不允许生成占位光谱文件并计为成功。
3. 不允许为了提高成功率而降低数据真实性标准。
4. 不确定是否成功时，应按失败处理。
5. 宁可少采，也不能采集伪数据。
6. 所有失败样本必须记录原因。
7. 所有跳过样本必须记录原因。
8. 所有成功样本必须可追溯到原始数据源。

对于 `.jdx` 文件，必须满足：

1. 文件非空。
2. 文件不是 HTML。
3. 文件至少包含以下 JCAMP-DX 标记之一：

   - `##JCAMP-DX=`
   - `##TITLE=`
   - `##DATA TYPE=`

4. 如果内容包含 `<html`、`<!DOCTYPE html>`、`<body` 等 HTML 标记，应判定为失败。
5. HTML 页面只能保存为 `.html`，不能保存为 `.jdx`。

---

## 5. 采集脚本通用要求

所有采集脚本必须支持：

1. 断点续跑。
2. 日志记录。
3. 失败记录。
4. 跳过记录。
5. Ctrl+C 安全退出。
6. 重试机制。
7. 命令行参数。
8. 小样本测试。
9. 输出统计汇总。
10. 每处理一个样本后立即保存进度。

所有采集脚本必须避免：

1. 未校验即保存文件。
2. 未命中却记为成功。
3. 把空文件记为成功。
4. 把错误页面记为成功。
5. 大规模运行前没有小样本验证。
6. 页面结构异常时继续盲目重试。
7. 使用不可复现的手动操作作为默认流程。

---

## 6. 调试原则

对于任何采集脚本，如果出现以下情况：

- 成功率异常低；
- 大量 skipped；
- 命中数与成功数严重不匹配；
- 页面解析失败；
- 输出文件格式不可信；
- 运行时间异常；
- 网站返回空白页、错误页或免责声明页；

必须立即停止扩大运行。

正确流程是：

1. 保存失败样本。
2. 保存页面 HTML。
3. 保存截图。
4. 记录当前 URL、页面标题、页面类型。
5. 分析真实页面结构。
6. 修复解析逻辑。
7. 用少量样本重新测试。
8. 成功后再扩大到 100、1000 或更大规模。

禁止在原因未明时继续扩大运行。

---

## 7. NIST 当前规则

NIST 是 Phase 1 的最高优先级数据源。

当前目标：

**修复并验证真实 JCAMP-DX 下载链路。**

NIST 成功标准：

1. 能够从 NIST WebBook 页面解析真实 JCAMP 下载链接。
2. 下载结果不是 HTML。
3. `.jdx` 文件通过 JCAMP-DX 标记校验。
4. 每个成功文件都有对应 metadata。
5. 失败和跳过均有明确原因。
6. 先通过前 10 个 CID 测试，再跑 100 个，再跑 1000 个。

NIST 禁止事项：

1. 不允许把 NIST 普通 HTML 页面保存成 `.jdx`。
2. 不允许把页面锚点误当作下载链接。
3. 不允许跳过校验直接保存。
4. 不允许在 10 样本测试失败时扩大运行。
5. 不允许清理旧伪 `.jdx` 文件，除非用户明确确认。

---

## 8. SDBS 当前规则

SDBS 当前不作为 Phase 1 的主力数据源。

当前状态：

1. 脚本曾经能够搜索部分 CAS。
2. 大规模随机测试中覆盖率和成功率极低。
3. 结果页、详情页和光谱链接解析仍不可靠。
4. 当前不应继续扩大运行。

SDBS 当前目标：

**只针对少量已命中的 CAS 样本调试页面结构。**

SDBS 成功标准：

1. 能区分 disclaimer、home、search、result list、compound detail、zero hit、error。
2. 能稳定进入 compound/detail 页面。
3. 能识别真实 IR / Raman / UV-Vis 入口。
4. 能保存真实光谱或结构化峰表。
5. 能解释每个测试样本为什么成功或失败。
6. 不能把 0 hit 页面计为 result page。
7. 不能把占位文件计为成功。

SDBS 调试流程：

1. 从已命中 CAS 样本中选 5 个。
2. 保存搜索页 HTML。
3. 保存结果页 HTML。
4. 保存详情页 HTML。
5. 保存截图。
6. 分析真实链接结构。
7. 修改解析逻辑。
8. 仅运行这 5 个样本测试。
9. 通过后再扩大到 20、100、1000。

SDBS 禁止事项：

1. 禁止继续盲目跑 1000 或全量。
2. 禁止使用“寻找第一个可见输入框”的泛化逻辑作为最终方案。
3. 禁止把搜索页或 home 页当作结果页。
4. 禁止没有真实光谱时写成功 metadata。
5. 禁止把 GIF、HTML、占位文件混淆为结构化光谱数据。

## 9. CAS 缓存规则

NIST 和 SDBS 均可能依赖 CAS。

应建立或复用共享缓存：

- `cache/cid_cas.csv`

建议字段：

- `cid`
- `cas`
- `source`
- `status`
- `updated_at`
- `message`

规则：

1. 已查询过的 CID 不应重复请求 PubChem。
2. 无 CAS 的 CID 也应缓存。
3. 查询失败和 CAS 不存在必须区分。
4. 缓存写入必须兼容 Windows。
5. 写入时应避免文件锁导致程序崩溃。
6. `os.replace` 失败时应自动重试。
7. 单个缓存写入失败不应导致整个采集任务崩溃。
8. 如果缓存被 Excel、VS Code 或其他进程占用，应记录错误并安全处理。

## 11. 重要脚本说明

### `scripts/build_master_in_order.py`

用于按顺序从 PubChem 构建 compound master。

当前阶段优先使用顺序采样结果。

### `scripts/build_master_stratified_random_sampling.py`

用于分层随机抽样构建大规模 compound master。

当前阶段暂不作为默认输入。

### `scripts/clean_master.py`

用于清洗 master，主要去除：

- 同位素记录
- 多组分记录

### `scripts/harvest_nist.py`

用于从 NIST WebBook 获取光谱数据。

当前最高优先级任务是确保真实 JCAMP-DX 下载。

### `scripts/harvest_sdbs.py`

用于从 SDBS 获取光谱数据。

当前任务是小样本页面结构调试，不是扩大运行。

## 12. Codex 工作方式

请把自己当成 SpectraDB 项目的开发者，而不是代码生成器。

每次接受任务时，应遵循：

1. 先阅读 `AGENTS.md`。
2. 再阅读 `TODO.md`。
3. 再阅读 `CHANGELOG.md`。
4. 只阅读相关代码。
5. 不扫描大数据目录。
6. 先分析问题。
7. 给出最小修改方案。
8. 修改代码。
9. 运行小规模测试。
10. 如果测试失败，自行定位并继续修复。
11. 测试通过后再建议扩大规模。
12. 更新 `CHANGELOG.md`。
13. 更新 `TODO.md`。
14. 输出中文总结。

不要频繁等待用户确认，除非涉及：

- 删除数据；
- 覆盖已有 raw 数据；
- 改变项目方向；
- 大规模重构；
- 新增重量级外部依赖；
- 长时间全量运行；
- 运行可能消耗大量时间或网络资源的任务。

## 14. Git 与归档规则

默认不提交以下内容：

- `raw/`
- `cache/`
- `logs/`
- `archive/`
- 大型数据文件
- 临时调试文件
- 浏览器截图
- HTML 页面缓存
- `.jdx` 大规模输出

`master/` 默认应保留在项目中，除非用户另有要求。

归档旧数据时：

1. 不删除文件。
2. 移动到 `archive/YYYY-MM-DD/description/`。
3. 更新 `.gitignore`。
4. 更新 `.codexignore`。
5. 输出移动清单。
6. 不移动 `master/`、`scripts/`、`README.md`、`AGENTS.md`、`TODO.md`、`CHANGELOG.md`。

## 15. 推荐测试流程

任何采集脚本修改后，默认测试顺序：

1. 静态检查。
2. 前 10 个样本测试。
3. 检查输出文件真实性。
4. 检查日志。
5. 检查 checkpoint。
6. 汇总成功、失败、跳过数量。
7. 分析失败原因。
8. 决定是否扩大到 100。
9. 100 通过后再考虑 1000。
10. 1000 通过后再考虑更大规模。

禁止跳过小样本测试直接全量运行。

## 16. 当前最高优先级

当前最高优先级如下：

1. 更新 `AGENTS.md` 和 `TODO.md`，明确 Phase 1 策略。
2. 将旧测试产物移动到 `archive/`，但不得删除。
3. 更新 `.gitignore` 和 `.codexignore`，忽略 `archive/`、`raw/`、`cache/`、`logs/`。
4. 建立、修复并验证 `scripts/harvest_nist.py` 的真实 JCAMP-DX 下载。
5. 建立并稳定 `cache/cid_cas.csv`。
6. 用顺序采样 master 运行 NIST 前 10 个样本测试。
7. NIST 通过后再扩大到 100 和 1000。
8. SDBS 暂停大规模运行，只保留 5 个命中 CAS 样本调试。

## 17. 长期路线图

### Phase 1：数据获取基础设施

目标：

- 稳定 NIST。
- 调试 SDBS。
- 建立 CAS 缓存。
- 建立数据真实性校验。
- 建立日志、checkpoint 和归档流程。

### Phase 2：数据规模扩展

目标：

- 扩大顺序采样采集规模。
- 重新启用随机 / 分层随机采样。
- 比较顺序采样与随机采样覆盖率。
- 建立数据质量统计报告。

### Phase 3：多模态配对

目标：

- 按 CID、InChIKey、CAS 统一关联 IR、Raman、UV-Vis。
- 建立单模态、双模态、三模态数据子集。
- 统计模态覆盖率。

### Phase 4：模型原型

目标：

- 建立光谱预处理流程。
- 训练基础识别模型。
- 评估 Top-k、拒识、开放集识别。
- 初步进行可解释性分析。
